# Lake County Contract Extraction & Analysis Pipeline — Project Summary

**Last updated: 2026-07-21 (session handoff — porting chat-app work to a new session)**

## Overview

Tiered extraction and analysis pipeline for 389 Lake County procurement contracts: structured-field extraction (Claude tool-use + vision), a unified `contracts_master` table with computed date/status columns, semantic clause retrieval (ChromaDB + Voyage embeddings), and a working SQL/vector query router with a Streamlit chat UI aligned to a specific stakeholder walkthrough deck ("slide3_v5", external to this repo). **Full corpus extraction is DONE.** This is the primary handoff doc — read this first.

## Architecture

| Tier | Doc Types | Count | Extraction Method | Output Table |
|------|-----------|-------|-------------------|--------------|

123 Tier 1/1.5 + 226 Tier 2/4 = 349 documents processed (40 excluded). Full rationale for both descoping decisions (Tier 3 rate-adjustment mini-pipeline, 16 oversized docs) in `tiered_extraction_approach.md`.

### Document pipeline
- **Digital-text docs**: pypdf → `[PAGE N]` markers → truncate to 60,000 chars → Claude tool-use.
- **Scanned docs**: PyMuPDF rasterizes pages (≤1568px long edge) → sent as image blocks → Claude tool-use. `MAX_VISION_PAGES = 35` (raised from 20 after the 16-oversized-doc exclusion removed the outliers that would've made a higher cap expensive).

### Extraction prompt (`extraction_system_prompt.txt`, 13 numbered rules)
Key rules beyond the original citation-mandatory fix (rule 2, still the strongest-worded rule — **has been accidentally reverted to a weak one-liner by externally-provided prompt updates at least twice this project; always diff before overwriting, never accept wholesale**):
- **Rule 10**: `rate_increase` vs `rate_schedule_expansion` for `modification_type`.
- **Rule 11**: `fee_structure` tie-breaker (`time_and_materials` only with explicit labor-rate + separate materials/expense clause; `rate_based` is the default).
- **Rule 12**: `effective_date` **IS the last signature date, mechanically** — not an interpretive question. (This was iterated through several versions this session: first "explicit clause overrides signature block," which caused a real bug on doc `25306` where the source doc's own logic said the opposite; settled on "always last signature date" per explicit user direction, with the ground truth updated to match on the docs where this mattered.)
- **Rule 13**: `signer_title` from the signature block specifically.

**modification_type enum** includes `rate_schedule_expansion` (schema + prompt both updated; a stale hardcoded `MODIFICATION_TYPE_ENUM` set in `extract_contracts.py` also had to be updated separately — same enum lives in two places, watch for drift).

### Database Schema (`contracts.db`)

1. **extractions** (123 rows) — one row per Tier 1/1.5 doc, 19×{field}_value/{field}_page columns, `modification_type`, `used_vision`, `page_truncated`, `page_citation_retry`, `fields_missing_page`, `modification_summary_issue`, cost tracking.
2. **existence_only** (226 rows) — Tier 2/4 identity records only (filename, contract_id, doc_role, tier, family_size, file_size_kb).
3. **rate_adjustments** (1 row) — leftover from the pre-descoping Tier 3 test (Ciorba Group/22128). Harmless, not actively used, **intentionally left in place** (also causes the one filename-collision noted below — not a bug).
4. **contracts_master** (349 rows, rebuilt from scratch each run via `build_master_table.py`) — one row per document across all tiers. Extracted-feature columns NULL for Tier 2/4. **349, not the naive sum of 350** — `2024_09_16_Rate_Adjustment_for_Contract_22128_1.pdf` exists in both `existence_only` (Tier 4, post-descoping) and `rate_adjustments` (pre-descoping leftover); `INSERT OR REPLACE` on the `filename` primary key means the later one wins. Documented, not a bug — `build_master_table.py`'s final print already accounts for this and explains it if it recurs.

**Computed date/status columns on `contracts_master`** (pure post-processing over already-extracted prose, no re-extraction — logic lives in `date_utils.py`, shared with `compare_ground_truth.py` so parsing doesn't drift between the two):
- `parsed_effective_date`, `parsed_expiration_date` (TEXT, `YYYY-MM-DD` or NULL)
- `computed_status` (`active` / `expired` / `expired_but_renewal_on_file` / `not_yet_effective` / `unknown`) — `expired_but_renewal_on_file` fires when the base agreement's own dates say expired but a `renewal_letter`/`extension` exists in `existence_only` for the same `contract_id`; kept distinct from both `active` and `expired` deliberately.
- `renewal_action_needed` (0/1): expiring ≤90 days AND `renewal_mechanism` is `auto_renew_unless_cancelled` or `option_to_renew`.
- `parsed_notice_period_days`, `parsed_escalation_pct`.
- **Guard**: if `parsed_expiration_date < parsed_effective_date` (seen in practice — a source doc's own year typo), `parsed_expiration_date` is nulled rather than trusted — don't compute a confident status off an internally-inconsistent date.
- `renewal_confidence` (`high` / `moderate` / `low` / NULL) + `renewal_confidence_basis` (TEXT, human-readable) — added 2026-07-20, see Known Fixes #8. Filename-only heuristic (`renewal_confidence.py`) over each contract_id's renewal_letter/extension documents in `existence_only`. Populated for ALL contracts_master rows (any tier) whose contract_id has a renewal doc on file, not just `expired_but_renewal_on_file` rows — deliberately does NOT overwrite `computed_status`, kept as a separate, auditable column. `query_router.py`'s schema description was updated to include it, with guidance that "active" queries can reasonably include `high`/`moderate` renewal_confidence rows alongside true `active` ones.

### Vector store (`chroma_db/`, collection `contract_clauses`)
1,457 chunks across all 123 Tier 1/1.5 docs. Section-based chunking when ≥3 `SECTION N. TITLE` headers detected, page-based fallback otherwise. Scanned docs get a one-time vision transcription pass (`transcribe_scanned_doc()` in `build_vector_store.py`) before chunking. Modification/amendment docs get an extra `modification_summary`-tagged chunk.

**Resilience fixes made this session** (a 123-doc build crashed on a transient network error at doc 109/123 and the original code lost all 108 completed documents' work, since everything was held in memory and written once at the very end):
- `transcribe_scanned_doc()` now retries transient network errors (`httpx.ReadError`/`ConnectError`/`RemoteProtocolError`, `anthropic.APIConnectionError`) with exponential backoff.
- `main()` now embeds+writes **per document**, not in one batch at the end.
- `--resume` flag: skips filenames already in the collection, so a crash only costs the one in-flight doc. Usage: `py build_vector_store.py --folder ./contracts/contracts --db contracts.db --resume`.
- Transcription uses `client.messages.stream(..., max_tokens=64000)` (was a non-streaming `max_tokens=4000`, then briefly 16000 — both silently truncated long scanned docs; streaming removes the ceiling). Still prints a warning if `stop_reason == "max_tokens"` even with streaming, as a safety net.

### Query router (`query_router.py`)
One Claude tool-use call decides SQL-path vs vector-path and produces the query in the same call; a second call synthesizes the final answer with citations.
- **SQL path**: Claude writes a read-only `SELECT` against `contracts_master` (validated: single statement, `SELECT`-only, no dangerous keywords). Schema description given to the model explicitly says to use the computed columns (`parsed_*`, `computed_status`, `renewal_action_needed`) for date/status filtering, never the raw prose fields.
- **Vector path**: embeds the question via Voyage, queries ChromaDB top-k, synthesizes an answer from retrieved chunks with `(filename, section, p.N)` citations.
- **Bug found and fixed this session**: SQL synthesis was hallucinating wrong conclusions from correct SQL results (e.g. "no contracts expiring in 90 days" despite 3 correctly-filtered rows) because the model tried to re-derive date math itself without knowing today's date. Fixed: today's date is now passed explicitly into the synthesis prompt, with an instruction to trust the SQL's already-correct filtering rather than re-verifying it.
- **Graceful failure fixed 2026-07-21**: `route_question()`'s `tool_choice` was `{"type": "any"}`, which *forces* a tool call — the model could never actually decline to classify a nonsense/off-topic question, so it would silently pick a tool anyway (observed: "asdfjkl random nonsense" got routed to SQL with a generic "show all contracts" query). Changed to `{"type": "auto"}` plus an explicit system-prompt instruction to respond in plain text instead of calling a tool for incoherent input — this is what actually makes `RouterClassificationError` fire for genuine gibberish. Also added: a single bounded SQL retry on execution error (`run_sql_with_retry()`, asks the model to fix its own query once, given the error, before falling back), a `VECTOR_DISTANCE_THRESHOLD = 1.0` cutoff on vector search (empirically calibrated: "indemnification clause" → best distance 0.926; "asdfjkl random nonsense" → best distance 1.085 — chunks worse than 1.0 are dropped rather than fed to synthesis), and a top-level `try/except` in `answer_question()` so nothing unhandled escapes. Every fallback string is routed through `_as_answer(text, stream)` — worth knowing about if you add a new failure path, since `st.write_stream()` raises `StreamlitAPIException` on a bare `str` (only accepts a generator/iterable-of-chunks), and every fallback site needs this wrapping consistently, not just the ones someone remembers by hand (this was a real bug caught during testing, not hypothetical).
- **Visualization added 2026-07-21**: `should_visualize(columns, rows)` returns true when >3 rows OR the 2-column (label, number) chart-candidate shape (`is_chart_candidate()`) — when true, `synthesize_sql_answer()` switches to a one-line-headline prompt instead of full per-row prose+citations, since the table/chart carries the detail. Note the table itself renders in `chat_app.py` for *any* multi-row result (>1 row, not >3) — that's intentionally more generous than the >3 rule, which only governs the synthesis prompt mode; a 2-row list still gets a visible `st.dataframe` alongside the full prose.
- **`query_via_sql` tool description bug found and fixed 2026-07-21**: it said "Always include filename in the SELECT list," which the model correctly followed even for GROUP BY/count queries -- producing a 3-column result (label, count, filename) that silently failed `is_chart_candidate()`'s 2-column check every time, so no aggregate query could ever render as a chart. Fixed by adding an explicit GROUP BY exception to the tool description: omit filename, select only the grouping + aggregate columns. Found this by actually checking `r['is_chart']` on a "What's the status of our contracts?" test call before assuming the charting feature worked -- it silently returned `False` until this was fixed.
- **Intent-normalization block added 2026-07-21**: `INTENT_NORMALIZATION` in `query_router.py`, inserted into `ROUTER_SYSTEM_PROMPT` before the schema description -- maps common vague phrasings ("what's the status of our contracts," "who are our vendors," etc.) to the correct GROUP BY/WHERE shape. Only added to the primary routing prompt, not `_regenerate_sql()`'s retry prompt (that path is narrowly for fixing a broken query given its own error, not full re-classification).

### Streamlit chat UI (`chat_app.py`)
Minimal chat interface, shows routing path + reasoning in an expander for debugging, plus a sidebar "Dev mode" checkbox that surfaces the same `timings` dict inside that expander. Launch: `py -m streamlit run chat_app.py` (bare `streamlit` isn't on PATH — see note below). `.claude/launch.json` has a `chat-app` config for the preview tool.
- **Speed fixes 2026-07-21**: profiled first (CLI test query: routing LLM call 3.53s, SQL execute 0.002s, synthesis LLM call 6.69s, total 10.27s — the two LLM calls are ~99.95% of latency, synthesis alone is 65%). Anthropic/Voyage clients were being re-constructed on every single query inside `answer_question()`; `chat_app.py` now builds them once via `@st.cache_resource`-wrapped factories and passes them in (kept optional in `answer_question()`'s signature so the CLI path still works standalone without Streamlit). Synthesis LLM calls now stream via `client.messages.stream(...)` + `st.write_stream()` instead of blocking for the full response — this is the fix that actually matters given synthesis was the biggest chunk of latency; client caching saves a real but comparatively tiny ~0.04s/query.

## Known Fixes Made This Session (chronological, for context on why things look the way they do)

1. **`compare_ground_truth.py` scoring bugs** (was under-crediting real accuracy): blank any value starting with `"not_stated"` on both sides (was exact-match only), word-overlap fallback for prose fields, M/D/YYYY date recognition, and a bug where the literal string `"none"` was being treated as a null-marker even though it's a valid `renewal_mechanism`/`annual_price_escalation_type` enum value.
2. **`extract_contracts.py` `_modification_summary_issue()` bug**: silently allowed `modification_type` to write `NULL` to the DB when the model collapsed `modification_summary` to a bare string `"not_applicable_doc_type"` instead of the required object shape — the bare-scalar case was assumed to be "already caught elsewhere," but the elsewhere-check (`_fields_missing_page`) exempts that exact sentinel value via `NULL_PAGE_OK_VALUES`, so the collapse went undetected. Fixed with an explicit check.
3. **16 oversized Tier 1/1.5 docs excluded** (moved to Tier 2): 10 scanned docs >50 pages, 6 digital docs >60 pages (up to 405 pages) — cost/effort exclusion, not misclassification, same reasoning as the pre-existing lease exclusion. `MAX_VISION_PAGES` raised 20→35 afterward since the true outliers were gone.
4. **Tier 3 (rate_adjustment mini-pipeline) descoped to Tier 4**: sample too thin (7 docs → 3 vendor relationships), per-role rate logic needed more domain input. Files preserved with `RETIRED_` prefix, not deleted.
5. **`build_vector_store.py` transcription truncation** (see "Resilience fixes" above) — three iterations: 4000→16000 tokens (helped but still truncated long docs) → streaming with 64000 (removed the ceiling) → added retry + incremental writes + `--resume` (crash resilience).
6. **`date_utils.py` two parser gaps** (found via a `computed_status` audit, not guessed): `normalize_date()` didn't handle dash-separated dates, 2-digit years, or day-first "DD Month YYYY" format (25 real dates were failing silently into `unknown`); `parse_duration()` only matched digit-form durations, not spelled-out legal prose ("one year from effective date"). Both fixed; `computed_status` distribution shifted meaningfully after rebuild (`unknown` 95→82, `active` 8→14, `expired_but_renewal_on_file` 4→9) — confirmed genuine signal, not noise.
7. **`query_router.py` SQL-synthesis date-hallucination bug** (see "Query router" above).
8. **`renewal_confidence` filename heuristic built 2026-07-20** (new `renewal_confidence.py`, wired into `build_master_table.py` and `query_router.py`'s schema): turns the flat `expired_but_renewal_on_file` bucket into a tiered signal (high/moderate/low) using only renewal_letter/extension filename patterns — no new extraction, no API calls. **Important correction to the task's own working assumption**: the task brief hypothesized Lake County's fiscal year runs Dec 1–Nov 30 uniformly and that a filename suffix like `"25_26"` maps to that window. Opened 6 actual renewal letters to check before trusting this (`22125_1`, `18018`, `21003`×2, `22128_1`, `22162`) — **every one showed the real coverage period anchored to the contract's own anniversary date instead** (observed start months: Feb, Mar, Aug, Oct — not Dec). What *did* hold across all 6: the two calendar years named in the filename always match the two calendar years the real period spans. Built Tier A (`high`) on that weaker, verified claim only — "today's year falls within [first_year, second_year]" — not a specific month/day boundary. Also found a real 2-digit/4-digit mixed range pattern in the corpus (`"2022_23"`, contract 18194) and added a third regex for it. Result: of the 9 `expired_but_renewal_on_file` rows, 7 (contract_ids 22125, 22128, 22162, 23018, 23173) get `renewal_confidence = 'high'` with a specific covering year-range; 2 (18194, 22060) correctly stay `low` — their most recent renewal filename (`2022_23`, `2024_2025`) is genuinely stale relative to today (2026-07-20), not a parser gap. Cross-checked 22128 against ground truth's rate-adjustment row (`expiration_date` 2025-10-18, matching the *pre-renewal* base period exactly) as an independent sanity check. **Side finding, not fixed**: `23018_Department_Renewal_26_27.pdf` — an even more current renewal filename than the one used — is classified `doc_role = unclassified` upstream and never made it into `existence_only`, so this heuristic can't see it (didn't change 23018's tier here since it was already `high`, but is a real gap in the unclassified-docs triage, flagged not chased).
9. **`computed_status` split: `expiration_not_stated` carved out of `unknown` (2026-07-21)**: the user flagged that a flat `unknown` label reads as a pipeline error to a deal-team audience when most of it is actually a clean fact (no expiration date stated). Investigated the full 37-row `doc_role='base_agreement'` subset of the old `unknown` bucket before deciding this was a real split, not just a relabel: 20 were vision-extracted (model looked at every page image, genuinely found no expiration — trustworthy), 12 were digital-extracted with the effective date found fine in the same pass (also trustworthy), and **5 were an actual extraction gap**, not genuine absence — opened the source PDFs and found 4 (`25154`, `25162`, `26115_3` Vastek, plus `25158` from an earlier session) share one DocuSign-template signature page where the filled-in date/name are form-field values invisible to pypdf's plain-text extraction (confirmed by rendering the page: the values are clearly printed, pypdf just returns blank underscores), and 1 (`25168`) has 17 of 19 pages return zero characters yet is still classified digital, never triggering the vision fallback — essentially blind to almost the whole document. **User explicitly declined to fix those 5** ("small enough and not worth it") — they still fall into `unknown`, which is now correct and honest specifically *because* `unknown` was redefined to mean "effective date itself unresolved" (all 5 have `effective_date_value = 'not_stated'` too, so they land there naturally with no special-casing needed). Fix: in `build_master_table.py`'s `_compute_date_columns()`, the branch that used to set `status = "unknown"` when `parsed_eff` was known but `parsed_exp` was `None` now sets `"expiration_not_stated"` instead; the other `unknown` branch (`parsed_eff is None`) is untouched.
